# Documentation Style Guide

This guide defines conventions for all MEDRE documentation. Follow these rules
when writing or editing any file under `docs/`.

## One Authoritative Location Per Topic

Every topic has exactly one authoritative document. Do not create parallel
descriptions of the same concept in multiple files.

- **Spec semantics** go into `docs/spec/` pages.
- **Operator procedures** go into `docs/ops/` pages.
- **Developer references** go into `docs/dev/` pages.

If you are unsure where a topic belongs, ask before writing.

## No One-Off Contract Documents

Do not create new numbered contract files under any directory. The old
`docs/contracts/` system has been replaced by the consolidated `docs/spec/`
tree. New normative content goes into the appropriate existing spec page.

If a spec page is missing, create it by following the existing naming and
structure conventions — not by adding a standalone contract.

## No Internal Planning-Cycle Vocabulary

The following terms are internal development vocabulary and are forbidden in
**all durable artifacts**:

- "tranche" (use "release scope", "work package", or "implementation phase")
- "boulder" (use "task" or "work item")
- "sprint" (use "development cycle")
- "ponytail" (an internal development skill name; describe the underlying idea
  plainly, e.g. "minimal change" rather than referencing the skill)

Durable artifacts include: documentation under `docs/` (including
`docs/spec/`, `docs/ops/`, `docs/dev/`, `docs/schemas/`, and `docs/changes/`),
source code comments and docstrings under `src/`, test names, test method
names, test comments, test docstrings, test filenames, example configs and
scripts under `examples/`, branch names, new commit messages, and
agent-facing prompts (including `AGENTS.md` and skill instructions).

These terms describe temporary internal planning state and must not leak into
any artifact that persists beyond the planning cycle. The only files that may
contain them are the definitional ones that enumerate the forbidden list
itself (`tests/helpers/forbidden_terms.py`), the enforcer test that names
the terms while checking for them
(`tests/test_docs_no_internal_planning_language.py`), and the style guide
that documents the ban (this page).

Historical git commit messages are preserved as-is — this policy does not
rewrite existing history. New commit messages must follow the stricter rule.

## No Stale Generated Headers

Do not add "Generated by …" or "Auto-generated" headers to documentation files
unless the content is actually generated by tooling (e.g., JSON Schema files
produced by a generation script). Hand-written documentation must not carry
false generated headers.

## RFC 2119 Keywords

The terms **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**, and **MAY**
(RFC 2119) are reserved for normative specification files under `docs/spec/`.
They must not appear in operator or developer documentation. Use plain language
instead:

- Instead of "You MUST run `medre smoke` before bridging", write
  "Run `medre smoke` to validate your configuration before bridging."

## Source Audits

Source audit notes are evidence of review, not normative authority. They belong
in `docs/dev/source-audits.md` or an appendix — never as standalone documents
in the spec tree.

## Adding New Documentation

1. Identify the correct directory (`spec/`, `ops/`, `dev/`).
2. Check that no existing file already covers the topic.
3. If extending an existing file, add a new section with a clear heading.
4. If creating a new file, follow the naming conventions of its siblings.
5. Do not create new top-level directories under `docs/` without explicit
   approval in a change fragment.

## Markdown Conventions

- Use ATX-style headers (`#` not underlines).
- Wrap long lines at 88 characters where practical.
- Use fenced code blocks with language tags.
- Use relative links between documents.
- One blank line before and after headers.
